## はじめに

実践的な検討を Codex Cli で行っておうとしているのですが、せっかくなので 2025-10 現在の AI コーディングがどんなものなのかを記録しておこうと思い立ちました。

なぜ Codex Cli なのかというと、[ひとつ前の記事](/blog/202510/26/codex-cli-first-impression)に書いた通り今私が契約しているのが Codex Cli だからです。
(本稿で検討していくのは実際の業務で用いる想定のものですが、個人 blog に書いていることからわかる通り、業務時間外のプライベートなリソースを使っています)

## 何をするか

ウェブアプリケーションにおける年月や時間の取り扱いについて考えていきます。
複数のタイムゾーンが登場すると問題が起こりがちなのは経験的に想像できるかと思いますが、そういった問題に対応するベストプラクティスを見つけられれば良いかなと考えています。

前節で触れた通り私が現在かかわっている業務に適用する想定の検討であり、一般化したものではありません。具体的には次のようなことを前提にしています:

- サーバーサイドは Spring Boot
- Spring Boot アプリケーションサーバ、RDBMS の動作するタイムゾーンは UTC
- 業務を行うのは JST。ただし将来的には Spring Boot をマルチテナント化してテナントごとにタイムゾーンが変えられるようにしたい

この件については、これまでもいろんなプロジェクトで問題になってきたのでおおよそ私の中では答えがあるのですが、それが本当に良い方法なのか、ということを考え直してみたいです。
ちなみに、私が今までとってきた解決策は、下記の記事に書いたように、 `Clock` Bean で業務に用いるタイムゾーンを持った時刻情報をインジェクションすることです。

- [Spring Boot 2.1 でテスト時 @Bean を挿げ替えたかった](/blog/201905/31/spring-boot-bean-on-testing/)

また、そもそもこのタイムゾーン問題をプロジェクトメンバーに理解してもらうために何かいい例題が無いか、というのもオプションで考えたいと思っています。
毎回この仕組みを導入するのに一番苦労するのが、プロジェクトメンバーに問題を理解してもらうことなので。
